# 遇到的问题与解决方案

## 1. 复习计划页面缺少任务展开和考察功能

**问题描述**：任务管理页面可以展开查看知识点并发起考察，但复习计划页面没有这个功能，两个页面体验不一致。

**原因分析**：复习计划页面只渲染了任务基本信息，缺少展开/收起交互和知识点展示、考察按钮。

**解决方案**：
- 在复习计划渲染逻辑中添加任务卡片的展开/收起功能
- 展开后展示关联的知识点列表
- 添加「开始考察这个任务」按钮，点击后跳转到考察页面并传入任务索引

**涉及文件**：
- `static/app.js` — 复习计划页面渲染逻辑

---

## 2. 知识点和任务关联不上

**问题描述**：自动关联算法只能匹配字符串包含关系的知识点，语义相关但没有字面包含关系的知识点无法匹配。

**原因分析**：匹配规则是严格的字符串包含（`kp in task` 或 `task in kp`），要求3字以上完全匹配。例如「快速排序算法」和「快排」就关联不上。

**解决方案**：
- 优化匹配逻辑，从严格包含改为关键词评分匹配
- 提取任务名称中的关键词（过滤停用词），与知识点名称做关键词匹配
- 得分超过阈值即视为匹配，支持模糊匹配
- 优先使用强关联表 `task_knowledge_points` 中的关联关系

**涉及文件**：
- `web_app.py` — `_auto_link_knowledge_to_tasks` 函数、`quiz_start` 函数

## 3. 考察结束生成总结失败（显示"请求失败"）

**问题描述**：点击「结束考察」后，前端页面显示「请求失败」，无法生成总结。

**原因分析**：`endQuiz()` 函数用 `res.json()` 解析响应，但后端 `/api/quiz/answer` 接口返回的是 SSE 流式响应（`text/event-stream`），不是标准 JSON，导致解析失败进入 catch 分支。

**解决方案**：
- 将 `endQuiz()` 函数改为使用 `res.body.getReader()` 流式读取
- 与出题、答题时的流式读取逻辑保持一致
- 逐块解析 SSE data 事件，最终获取 `finalData.ended` 状态

**涉及文件**：
- `static/app.js` — `endQuiz()` 函数

---

## 4. 考察风格太温和，需要大厂压力面试感

**问题描述**：考察系统提示词是"严格但友好的老师"，提问方式温和，不符合用户想要的压力面试感觉。

**原因分析**：系统提示词人设定位为老师，考察规则偏向教学式引导，缺乏压力感。

**解决方案**：
- 人设改为「一线大厂资深技术面试官」，风格犀利、直接、不留情面
- 新增连环追问机制，答对了继续往深了挖
- 增加质疑式点评，即使答对也挑战
- 结束评价改为犀利定级，指出硬伤和与大厂要求的差距
- 新增场景设计题、手撕代码题、边界 case 题等更难的题型

**涉及文件**：
- `web_app.py` — `quiz_system_prompt` 提示词

## 5. 考察直接考复习内容原文，不是考知识点理解

**问题描述**：考察时把复习内容直接拼给 LLM，导致题目偏向原文记忆（如"XX的定义是什么"），而不是考察对知识点的理解和应用。

**原因分析**：
- 后端把学习记录的原始内容直接作为考察素材
- 系统提示词没有明确禁止考原文记忆，也没有指定题型方向

**解决方案**：
- 重构后端考察逻辑，从数据库直接查询结构化的知识点（`knowledge_points` 表）
- 将知识点以列表形式传递给 LLM，包含名称、类型、描述、遗忘紧急度
- 系统提示词明确：**严禁出考原文背诵的题目**
- 指定7种题型方向：原理理解、应用场景、对比分析、场景设计、排错分析、手撕代码、边界 case

**涉及文件**：
- `web_app.py` — `quiz_start()` 函数、系统提示词

## 6. 考察结束后无法返回到具体学习任务

**问题描述**：考察结束后只有「再来一轮」按钮，想回到任务管理/复习计划页面只能手动点导航，体验不连贯。

**原因分析**：
- 考察页面没有记录来源页面信息
- 结束页面缺少返回按钮

**解决方案**：
- 在发起考察时传递 `source_page` 参数（tasks / review-plan）
- 用 `currentQuizParams` 全局变量保存来源信息
- 考察结束后，如果有来源页面，显示「← 返回任务」按钮
- 点击返回按钮根据来源跳转到对应页面

**涉及文件**：
- `static/app.js` — `startQuizFromTask()`、`startQuizFromPlan()`、`backToTaskFromQuiz()`、`endQuiz()`、`submitQuizAnswer()`

## 7. 考察结果不会自动更新任务掌握程度

**问题描述**：考察结束后只是生成评价，不会根据答题情况更新任务的掌握程度、复习次数，考察和任务管理是割裂的。

**原因分析**：
- 考察系统是独立功能，没有和任务数据打通
- 掌握程度只能通过"标记已复习"手动提升，不能根据考察结果自动评估

**解决方案**：
- 后端新增 `_analyze_quiz_results()` 函数，从考察历史中提取每道题的结果（✅正确/⚠️不完整/❌错误），统计正确率
- 新增 `_calc_new_mastery()` 函数，根据正确率计算新的掌握程度：
  - 答题数 ≥ 3 且正确率 ≥ 85%：升一级
  - 正确率 50%-85%：保持不变
  - 答题数 ≥ 3 且正确率 < 50%：降一级
- 新增 `_update_task_after_quiz()` 函数，更新任务的掌握程度、复习次数、最后复习时间
- 前端新增答题统计卡片，展示：总题数、正确/不完整/错误数量、正确率进度条、掌握程度变化
- 考察结束时自动更新并展示结果

**涉及文件**：
- `web_app.py` — `_analyze_quiz_results()`、`_calc_new_mastery()`、`_update_task_after_quiz()`、`quiz_answer()`
- `static/app.js` — `buildQuizStatsHtml()`、`endQuiz()`、`submitQuizAnswer()`

## 8. 考察批改只有答案没有思考过程

**问题描述**：考察批改时直接给出答案和解析，用户被动接受，不知道正确的思考路径是什么，学习效果不好。

**原因分析**：系统提示词中只有【解析】板块，要求"详细解析过程和正确答案"，但没有明确要求用引导式的方式带着用户思考。

**解决方案**：
- 将【解析】拆分为【思路引导】+【正确答案】两个板块
- 【思路引导】要求像老师带学生解题一样，一步步推导：
  - 先分析这道题的核心考点是什么
  - 应该从哪几个角度入手分析
  - 候选人的思路哪里走偏了
  - 正确的推理路径是什么
- 【正确答案】放在思路引导之后，给出简洁明确的结论
- 在考察核心规则中新增一条强约束："批改时不要直接甩答案！要先用思路引导带着候选人一步步思考"

**涉及文件**：
- `web_app.py` — 系统提示词 `quiz_system_prompt`

## 9. 添加用户系统和 JWT 认证

**问题描述**：项目是单用户模式，没有用户系统，数据不隔离，无法支撑多用户使用，作为全栈项目不够完整。

**原因分析**：
- 数据库所有表都没有 user_id 字段，数据是全局的
- 没有注册登录功能
- 没有身份认证机制

**解决方案**：
- 新增 `users` 表，存储用户信息（用户名、密码哈希、邮箱、昵称等）
- 给所有业务表（tasks、daily_records、knowledge_points、task_knowledge_points）添加 `user_id` 字段
- 使用 JWT（JSON Web Token）实现无状态认证
- 密码使用 Werkzeug 的 `generate_password_hash` 哈希存储，不存明文
- 后端新增注册、登录、获取当前用户信息三个 API
- 实现 `before_request` 钩子，每次请求解析 token 并设置当前用户
- 采用「可选登录」模式：未登录使用默认用户（user_id=0），兼容旧数据
- 前端添加登录/注册模态框，token 存 localStorage
- 数据库迁移：使用 ALTER TABLE 为旧数据库添加 user_id 列，注意要在创建索引之前执行

**涉及文件**：
- `src/memory.py` — 数据库初始化、用户管理方法、所有操作方法加 user_id 过滤
- `web_app.py` — JWT 工具函数、注册/登录/用户信息 API、请求钩子
- `templates/index.html` — 登录注册模态框、侧边栏用户状态
- `static/style.css` — 用户系统相关样式
- `static/app.js` — 登录注册逻辑、token 管理

## 10. 数据库迁移顺序导致初始化失败

**问题描述**：给旧数据库添加 user_id 字段时，初始化报错 `no such column: user_id`。

**原因分析**：
- 先执行 `CREATE INDEX IF NOT EXISTS idx_xxx_user_id` 创建索引
- 再执行 `ALTER TABLE ADD COLUMN user_id` 添加列
- 但旧表没有 user_id 列时，创建索引会失败，而且是在 `executescript` 中，整个脚本事务回滚

**解决方案**：
- 调整顺序：**先 ALTER TABLE 添加列，再创建表和索引**
- 因为 `CREATE TABLE IF NOT EXISTS` 遇到已存在的表会跳过，所以先加列不影响
- 注意用 `try/except` 捕获 ` OperationalError`，因为列可能已经存在

**涉及文件**：
- `src/memory.py` — `_init_db()` 方法的执行顺序
